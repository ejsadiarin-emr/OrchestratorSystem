[QUESTION]: Is it better to add the initialization steps (pre-init & post-init) via JSON at the package manifest level or workload definition level (inside packages array)
- I'm leaning more on **workload definition level** because there may be packages that depend on each other, for example custom-app-1 and postgresdb-v7, then connecting the custom-app-1 to the postgresdb-v7 would happen on the post-init step defined in the workload definition level. If this is package manifest level, then it isn't feasible to have "custom" steps like this (e.g. connecting a db to an app to be installed) because the package manifest level is supposed to be general.
- Example of a workload definition level pre/post initialization steps: CustomApp1 Workload JSON:
```jsonc
{
	"name": "CustomApp1 Workload",
	"slug": "customapp1-workload",
	"description": "A workload that installs custom-app-1",
	"version": "2.0.0",
	"packages": [
		{
			"name": "custom-app-1-v2",
			"preInitSteps": [
				"COMMAND: to check if custom-app-1-v2 already exists",
				"COMMAND: ping orchestrator db to check if this is recorded in the specific agent record",
			],
			"postInitSteps": [
				"",
			]
		},
		{
			"name": "postgresql-v7",
			"preInitSteps": [
				"psql --version or any COMMAND to check if postgresql-v7 already exists ",
			],
			"postInitSteps": [
				"custom-app-1 connection string",
			]
		}
	]
	// OPTIONAL:
	"postWorkloadSteps": [
		"COMMAND: Final verification if all packages defined are installed successfully in the agent node"
		"COMMAND: curl or Invoke-WebRequest to ping orchestrator endpoint for success/fail status"
	]
}
```
- BUT [QUESTION] (on this Workload schema): would this dependency really occur? What if the custom-app-1 already handles this by having the installer media handle the db installation, db connection, etc.?
