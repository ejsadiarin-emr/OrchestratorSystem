# Orchestrator Control Plane — Context

The Orchestrator is the single control plane: it plans runs, evaluates policy, manages artifacts, and provides all operator-facing UI. Agents are headless — the Orchestrator is the only surface for visibility and control.

## Language

**Orchestrator**:
The central control plane that plans workload runs, evaluates risk, manages enrollment, and drives agents. Self-contained executable with embedded React UI.
_Avoid_: server, controller, master

**Workload Run**:
A single execution of a workload revision against one or more target nodes. The core runtime API surface uses `/api/workload-runs` exclusively.
_Avoid_: job, deployment, task (use "job" only in legacy deprecation context)

**Enrollment Token**:
A short-lived, single-use token generated by the Orchestrator. The operator creates one, then a remote node uses it (embedded in the downloaded agent.exe) to authenticate and join the fleet.
_Avoid_: auth token, API key

**Risk Level**:
A manifest-level classification (`low | medium | high`) evaluated by the Policy Engine. Signature verification warnings elevate risk to `high`. Risk status is displayed in the UI but does not block automatic updates.
_Avoid_: severity, priority, approval level

**Policy Engine**:
Evaluates risk level and workload constraints before a run proceeds. In PoC Phase 1, the update workflow is fully automatic: pre-check → detect risk → display status in orchestrator UI → proceed via pre-defined upgrade paths. No manual approval gate.
_Avoid_: rules engine, decision service

**Workload Definition**:
A versioned JSON document imported as part of a global workload file (2–3 workloads per PoC file). Declares what packages to install and in what order. Once published as a revision, content is immutable.
_Avoid_: workload config, deployment template

**Pre-Upgrade Action**:
A declarative requirement in a workload revision that must be satisfied before an upgrade proceeds — e.g., "backup database", "stop service". The Orchestrator interprets these during run planning.
_Avoid_: upgrade path, pre-condition, prerequisite

## Relationships

- **Orchestrator → Agent**: enrollment tokens, run assignments (SignalR), heartbeat/telemetry ingestion
- **Orchestrator → Artifact Catalog**: manifest retrieval for risk checks; artifact ingest via drag-and-drop/file picker UI
- **Orchestrator → Operator**: embedded web UI for all visibility and control (no agent UI)

## Example dialogue

> **Dev:** "Does the operator need to approve an update before it runs?"
> **Domain expert:** "No — the update workflow is fully automatic. The operator can view risk status in the UI and cancel before dispatch, but there is no manual approval gate."

> **Dev:** "How does an operator install agents on remote nodes?"
> **Domain expert:** "Generate an enrollment token in the Orchestrator UI, then open the provided URL on the target node to download a signed agent.exe. The token authorizes the agent to join. No scripts, no WinRM."

## Flagged ambiguities

- "Fleet" was used in early UI labels. Resolved: use "nodes" or "agents" in operator-facing terminology. Legacy fleet wording has been removed.
- "Job" was the legacy runtime API term. Resolved: all runtime APIs use `/api/workload-runs`. `/api/jobs` mutations return 410 Gone.

## CLI Command Surface (Draft)

The CLI (`di`) provides REST-parity workload command access for operators who prefer terminal workflows:

| Command | Purpose |
|---|---|
| `di workload-runs create` | Submit an install/update/rollback workload run |
| `di workload-runs status <runId>` | Fetch run summary and state |
| `di workload-runs cancel <runId>` | Cancel an active workload run |
| `di workloads list` | List workload definitions and revisions |
| `di nodes list` | List node health and registration |
| `di artifacts upload --file <path> --manifest <path>` | Ingest artifact + manifest |

CLI is the scripted surface; UI is the interactive surface. Both map to the same REST API.

## Orchestrator Self-Update (Draft)

The Orchestrator updates itself using a staged-swap pattern:

1. Download new orchestrator binary to staging location
2. Verify signature and hash
3. Create config snapshot (per config persistence contract)
4. Swap: rename current binary, move new binary into place
5. Restart service
6. On failure: restore from config snapshot and roll back to previous binary

This applies only to the Orchestrator node itself. Agent updates are dispatched as workload runs.

## Bounded Context

The Orchestrator owns:
- REST API and web UI (embedded React, self-contained executable)
- Run planning, scheduling, and dispatch
- Policy evaluation (risk level interpretation, pre-upgrade action enforcement)
- Agent enrollment (token generation, validation, mTLS channel establishment)
- Workload definition import (global JSON file, drag-and-drop in UI)
- Local artifact store management (upload, manifest validation, version tracking)
- Run monitoring and telemetry aggregation (SignalR status, OTel → Loki → Grafana)

It does NOT own:
- Artifact content or binary storage format (Artifact & Workload Catalog context)
- Execution on target nodes (Agent Runtime context)

## Invariants

1. An Agent cannot execute a run without a valid enrollment token.
2. Policy evaluation happens in the Orchestrator before a run is dispatched.
3. Workload definitions are immutable once published; updates create new revisions.
4. Update workflow is fully automatic — no manual approval gate. Operators can cancel but not approve.
5. Bootstrap is browser-based (download signed agent.exe + enrollment token), not WinRM push.
6. The Orchestrator UI is the sole operator surface; the Agent has no UI.
7. Running a workload from the Orchestrator UI is the primary Phase 1 demo goal.
